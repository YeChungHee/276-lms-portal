# 276홀딩스 사내 교육 포털 (Employee Learning Portal) 전체 계획서

> **문서 버전:** v1.0
> **기준일:** 2026-05-30
> **기술 스택:** Static HTML/CSS/JS (Frontend) + Google Apps Script (Backend API) + Google Sheets (DB)
> **범위:** 계정 요청·승인, 사원 로그인, 회사소개·교과서 학습, 퀴즈 응시, 점수 확인, 취약 교과 재학습, 관리자 통합 운영
> **선행 문서:** `온보딩_LMS_계획서.md` (v0.1), `Slack_알림_연동_계획서.md` (v1.0)

## 1. 개요 및 목표

본 계획서는 기존에 분산되어 있던 핀테크 교과서·OX퀴즈·관리자 대시보드를 **하나의 사원 계정 기반 교육 포털**로 통합·확장하기 위한 마스터 플랜이다.

기존 시스템은 "이름 입력"만으로 콘텐츠에 접근하는 공개형 구조였으나, 신규 포털은 **계정 발급 → 로그인 → 개인 학습 추적 → 취약점 보강**의 폐쇄형 LMS로 전환한다. 동시에 모든 파일명을 의미가 명확한 영문 체계로 재구성하여 유지보수성을 높인다.

| 목표 | 설명 |
|---|---|
| 통합 진입점 | `index.html`을 **사원용 로그인 페이지**로 전환, 모든 학습 경로의 단일 진입점화 |
| 계정 기반 추적 | 사원별 진도·점수·취약점을 계정 단위로 영구 추적 |
| 운영 자동화 | 계정 승인, 퀴즈 승인, Slack 알림을 관리자 화면 한 곳에서 처리 |
| 콘텐츠 관리 | 교과서·퀴즈·교육과정을 관리자가 직접 생성/수정 |
| 개인화 학습 | 취약 태그 진단 → 부족 교과 재학습 자동 배정 |

## 4. 사원용 페이지 (Employee-facing)

### 4-1. `index.html` — 로그인 페이지

포털의 단일 진입점. 첫 화면 타이틀은 `Welcome NewMember System`.

| 요소 | 설명 |
|---|---|
| 로그인 폼 | 로그인 아이디 + 비밀번호 입력 |
| Sign-Up 링크 | `signup.html`로 이동 (계정 미보유자) |
| 비밀번호 찾기 | 관리자 문의 안내 (1차 버전은 수동 재발급) |
| 첫 로그인 처리 | 임시 비밀번호 로그인 시 비밀번호 변경 페이지로 강제 이동 |

**인증 흐름:** 아이디/비밀번호 → Apps Script `action: login` → 검증 성공 시 세션 토큰(또는 사원ID) 반환 → 브라우저 메모리/`sessionStorage`에 보관 → `dashboard.html` 이동.

> ⚠️ Apps Script 환경의 보안 한계상 비밀번호는 **해시(SHA-256 + salt)** 로 저장하고, 평문 전송 최소화를 위해 HTTPS 배포를 전제한다. 토큰은 단순 만료형으로 운영한다.

### 4-2. `signup.html` — 계정 생성 요청

| 필드 | 검증 |
|---|---|
| 이름 (실명) | 2자 이상 |
| 휴대폰 | 숫자·하이픈 |
| 이메일용 아이디 | 영문+숫자, 4자 이상 |
| 희망 부서 | 영업/심사/기획/운영/AX 선택 |

제출 시 `account_requests`에 `상태: 관리자확인대기`로 저장. 동시에 관리자에게 Slack DM 발송(선택). 신청자에게는 "관리자 승인 후 Slack/이메일로 계정이 전달됩니다" 안내.

### 4-3. `dashboard.html` — 개인 학습 대시보드

로그인 후 메인 화면. 사원이 보는 모든 학습 활동의 허브.

| 영역 | 내용 |
|---|---|
| 진도 현황 | 전체 진도율, 주차별 완료 상태 (프로그레스 바) |
| 학습 메뉴 | 회사소개 / 교과서(핀테크·경제) / 부서별 모듈 |
| 퀴즈 메뉴 | 응시 가능 퀴즈, 응시 신청, 승인 상태 |
| 내 점수 | 퀴즈별 점수·등급, 추이 그래프 |
| 보강 학습 | 취약 태그 기반 재학습 과제 (배정 시 알림) |
| 공지/알림 | 관리자 공지, Slack 연동 상태 |

### 4-4. 학습 콘텐츠 페이지

| 페이지 | 내용 | 진도 추적 |
|---|---|---|
| `company-intro.html` | 회사 소개, 조직, FlowPay 서비스 구조 | 열람 완료 시 `progress` 기록 |
| `textbook-fintech.html` | 핀테크/SCF/BNPL·SNPL/전자어음/팩토링 | 챕터별 완료 기록 |
| `textbook-economy.html` | 교통·통신·금융 발달과 경제성장 | 챕터별 완료 기록 |
| `textbook-flowpay-platform.html` | 플로우페이 플랫폼 사용하기: 서비스 정의·자금흐름·신청 4단계·심사/약정/서명·FAQ | 챕터별 완료 기록 |

각 교과서는 챕터 완료 시 `action: progress`로 사원ID·모듈ID·완료일시를 저장한다. (기존 `학습진도` 시트 구조 확장)

### 4-5. 퀴즈 응시 흐름

```
대시보드 → 퀴즈 신청 (선행 학습 완료 검증)
   │  action: quiz_request → account 기반 저장 + 관리자 Slack DM
   ▼
관리자 승인 (admin.html)
   │  action: approve → 승인 + 사원 Slack DM
   ▼
사원: 대시보드 새로고침 → 승인 확인 → quiz-*.html 입장
   │  응시 → action: quiz_submit (사원ID·답안·점수·오답태그)
   ▼
채점 결과 + 취약 태그 산출 → 점수 페이지 반영
```

### 4-6. 점수 확인 & 부족 교과 재학습

점수 화면에서 사원은 퀴즈별 점수·등급과 **취약 태그 Top 3**를 확인한다. 취약점이 감지되면 `remediation_tasks`에 보강 과제가 자동 생성되고, 대시보드 "보강 학습" 영역에 노출된다.

| 보강 과제 항목 | 내용 |
|---|---|
| 보강 제목 | 예: "FlowPay SNPL 구조 재학습" |
| 원인 태그 | `FlowPay_SNPL`, `팩토링` 등 |
| 학습 자료 | 해당 교과서 챕터 링크 |
| 추가 문제 | 3~5문항 |
| 완료 기준 | 보강 퀴즈 80% 이상 |

## 5. 관리자 페이지 (`admin.html`)

기존 퀴즈 결과 조회 기능을 **포털 통합 운영 콘솔**로 확장한다. `ADMIN_KEY` 인증 유지(추후 관리자 계정 로그인으로 격상 권장).

### 5-1. 계정 관리

| 기능 | 설명 |
|---|---|
| 요청 목록 | `account_requests` 조회 (대기/승인/반려) |
| 계정 생성 | 중복 검증 → 로그인ID·이메일ID 확정 → 임시 비밀번호 발급 → `employees` 등록 |
| 커리큘럼 배정 | 부서별 커리큘럼 선택 |
| Slack 전달 | 계정 정보를 사원/멘토에게 DM 발송 |
| 계정 상태 관리 | 활성/비활성/초기화(비밀번호 재발급) |

### 5-2. 사원용 페이지 업데이트 기능

관리자가 사원에게 노출되는 콘텐츠·메뉴를 직접 갱신.

| 기능 | 설명 |
|---|---|
| 공지 등록 | 대시보드 상단 공지 작성 |

### 5-3. 교육과정 (Curriculum) 관리

| 기능 | 설명 |

| 커리큘럼 생성 | 이름·기간(4주 등)·대상 부서 정의 |

### 5-4. 교과서 생성/관리

| 기능 | 설명 |
|---|---|

## 6. 데이터 구조 (Google Sheets)

기존 3개 시트(`응답데이터`, `학습진도`, `퀴즈신청`)를 다음 11개 시트 체계로 확장한다.

| 시트 | 주요 컬럼 |
|---|---|
| `employees` | 사원ID, 이름, 로그인ID, 이메일ID, 휴대폰, 비밀번호해시, salt, 입사일, 부서, 멘토, 커리큘럼ID, 상태 |
| `account_requests` | 요청일시, 이름, 휴대폰, 이메일용아이디, 희망부서, 상태, 관리자메모 |
| `curriculums` | 커리큘럼ID, 이름, 기간, 대상부서 |
| `modules` | 모듈ID, 주차, 부서, 제목, 자료URL, 선행조건, 노출여부 |
| `progress` | 사원ID, 모듈ID, 상태, 완료일시 |
| `quizzes` | 퀴즈ID, 퀴즈명, 연결모듈ID, 합격기준%, 노출여부 |
| `quiz_questions` | 문항ID, 퀴즈ID, 질문, 정답, 해설, 진단태그 |
| `quiz_results` | 사원ID, 퀴즈ID, 점수, 정답수, 응시일시, 오답태그 |

## 7. 백엔드 API (Code.gs v3.0)

기존 `doGet`/`doPost` action 라우팅 패턴을 유지하며 action을 확장한다.

### 7-1. 사원용 action

| action | 메서드 | 설명 |
|---|---|---|
| `signup` | POST | 계정 요청 저장 |
| `login` | POST | 아이디/비밀번호 검증 → 토큰 반환 |
| `change_password` | POST | 첫 로그인 비밀번호 변경 |
| `get_dashboard` | GET | 사원 진도·점수·보강과제 통합 조회 |
| `progress` | POST | 모듈 완료 기록 (기존 유지·확장) |
| `quiz_request` | POST | 퀴즈 응시 신청 (계정 기반) |
| `check_approval` | GET | 승인 여부 확인 (기존 유지) |
| `quiz_submit` | POST | 답안 제출·채점·취약 태그 산출 |
| `get_scores` | GET | 사원 점수·취약점 조회 |

### 7-2. 관리자 action (ADMIN_KEY 필요)

| action | 메서드 | 설명 |
|---|---|---|
| `admin_requests` | GET | 계정 요청 목록 |
| `create_account` | POST | 계정 생성·임시비번 발급·Slack 전달 |
| `approve` | GET/POST | 퀴즈 신청 승인 (기존 유지) |
| `delete_request` | GET | 신청 삭제 (기존 유지) |
| `save_curriculum` | POST | 커리큘럼·모듈 저장 |
| `save_quiz` | POST | 퀴즈·문항 저장 |
| `admin_overview` | GET | 진도·점수·취약점 통합 통계 |

### 7-3. 신규 헬퍼

| 함수 | 역할 |
|---|---|
| `hashPassword(pw, salt)` | SHA-256 해시 |
| `verifyLogin(id, pw)` | 로그인 검증 |
| `analyzeWeakness(sawonId, quizId)` | 태그별 정답률 산출 → 취약점 갱신 |
| `createRemediation(sawonId, tags)` | 보강 과제 자동 생성 |
| `sendSlackDM(userId, text)` | Slack DM (연동 계획서 참조) |

## 8. 취약점 분석 로직

문항의 `진단태그`를 기준으로 산출한다.

| 기준 | 판정 |
|---|---|
| 카테고리 정답률 | 태그별 정답률 70% 미만 → 취약 |
| 반복 오답 | 동일 태그 2회 이상 오답 |
| 미완료 학습 | 선행 모듈 미완료 후 퀴즈 실패 |
| 응시 시간 | 비정상적으로 빠른 제출 |

진단 태그 예시: `핀테크기초`, `전자어음`, `팩토링`, `FlowPay_SNPL`, `고객신청`, `심사프로세스`, `CRM`, `콜드메일`, `운영데이터`, `AX자동화`.

## 9. Slack 연동

`Slack_알림_연동_계획서.md` (v1.0)의 설계를 그대로 채택·확장한다.

| 트리거 | 수신자 | 내용 |
|---|---|---|

| 퀴즈 승인 | 사원 | "응시가 승인되었습니다" |
| 보강 과제 배정 | 사원 | "보강 학습이 배정되었습니다" |
